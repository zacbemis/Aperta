# v1 ŌĆö Single-user PDF reader with cross-device page sync

## Goal

Prove the core experience: a user uploads PDFs, opens them in any browser, and
the current page stays in sync across devices.

## Scope

**In.** Phoenix 1.8 app, Postgres for metadata + sync state, MinIO
(S3-compatible) for PDF bytes, PDF.js viewer in the browser, `phx.gen.auth`
scaffold with a single seeded user, drag-and-drop upload, library index,
reader with realtime page sync via `Phoenix.PubSub`.

**Out (deferred).** Multi-user, sharing, annotations/highlights/notes, EPUB and
other formats, thumbnails, cloud hosting story, offline mode, full-text search.

## Architectural decisions

- **Storage.** PDF bytes live in MinIO under `documents/<document_id>.pdf`.
  Postgres stores metadata (`title`, `author`, `filename`, `page_count`, ŌĆ”) and
  sync state (`current_page`, `current_page_updated_at`). The `documents`
  schema carries a `format` column (`"pdf"` for v1) so non-PDF formats can be
  added later without a migration.
- **S3 client.** `ex_aws` + `ex_aws_s3`, configured to talk to MinIO in dev
  (`docker compose`) and real S3 later via runtime config.
- **Auth.** `mix phx.gen.auth` scaffold with the registration route disabled;
  `seeds.exs` creates the single user from env vars.
- **Reader.** The prebuilt PDF.js viewer (`pdfjs-dist`'s `web/viewer.html`)
  vendored to `priv/static/vendor/pdfjs/`, loaded inside an `<iframe>`. A
  LiveView JS hook bridges:
  - viewer ŌåÆ server: `pagechanging` / `pagesinit` events via `postMessage`,
    forwarded as `pushEvent("page_changed", %{page: n})` (debounced).
  - server ŌåÆ viewer: `handle_info({:page_updated, page, origin})` ŌåÆ `pushEvent`
    ŌåÆ `postMessage` to the iframe, which calls `PDFViewerApplication.page = n`.
  Origin IDs on broadcasts so a device ignores its own echoes.
- **Sync rule.** Last-writer-wins keyed by `current_page_updated_at`. Incoming
  writes older than the stored timestamp are dropped. Broadcast on
  `Phoenix.PubSub` topic `"document:#{id}"` so every connected device updates
  live.
- **Uploads.** LiveView `allow_upload/3` with `accept: ~w(.pdf)` and a
  reasonable `max_file_size`. Chunks are consumed to a temp file, uploaded to
  MinIO via `ExAws.S3.put_object/3`, and the `Document` row is inserted in the
  same `consume_uploaded_entry/3` callback.
- **Page count.** Captured client-side by PDF.js on first open
  (`PDFViewerApplication.pdfDocument.numPages`) and pushed back to the server
  so we never need a server-side PDF dependency in v1.

## Data model

```elixir
# priv/repo/migrations/<ts>_create_documents.exs
create table(:documents) do
  add :user_id, references(:users, on_delete: :delete_all), null: false
  add :format, :string, null: false, default: "pdf"
  add :title, :string, null: false
  add :author, :string
  add :filename, :string, null: false
  add :content_type, :string, null: false
  add :byte_size, :integer, null: false
  add :storage_key, :string, null: false
  add :page_count, :integer
  add :current_page, :integer, null: false, default: 1
  add :current_page_updated_at, :utc_datetime_usec

  timestamps(type: :utc_datetime_usec)
end

create index(:documents, [:user_id])
create unique_index(:documents, [:storage_key])
```

Contexts:

- `Aperta.Accounts` ŌĆö from `phx.gen.auth`.
- `Aperta.Library` ŌĆö `list_documents/1`, `get_document!/2`,
  `create_document/2`, `update_current_page/3` (enforces the LWW rule),
  `delete_document/2`.
- `Aperta.Storage` ŌĆö behaviour with `put/3`, `presigned_get_url/2`, `delete/1`.
  Default implementation `Aperta.Storage.S3`. Makes it easy to add a local-fs
  backend later for minimal self-hosts.

## Steps

Do **one step at a time**, run `mix precommit` (or at least `mix test`) after
each, stop for review before moving on.

### Step 1 ŌĆö Infra: MinIO in docker-compose

- Add a `minio` service to `docker-compose.yml` (ports `9000`/`9001`, volume,
  root user/password from env with dev defaults).
- Add a `minio-setup` one-shot service (or a `mix` task) that creates the
  `aperta-documents` bucket on boot.
- Update the README Quickstart so `docker compose up -d` now brings Postgres
  **and** MinIO.
- Verify by hand: open the MinIO console on `localhost:9001`.

### Step 2 ŌĆö Dependencies & config

- Add `{:ex_aws, "~> 2.5"}` and `{:ex_aws_s3, "~> 2.5"}` to `mix.exs`.
  (ExAws ships with hackney by default; we accept this for the S3 client only ŌĆö
  app HTTP stays on `Req`.)
- Configure ExAws in `config/runtime.exs` with `:host`, `:port`, `:scheme`,
  `:access_key_id`, `:secret_access_key` pulled from env, defaulting to the
  MinIO dev values in `config/dev.exs`.
- Add a `:bucket` config key under `:aperta, Aperta.Storage`.

### Step 3 ŌĆö `Aperta.Storage` behaviour + S3 impl

- Define `Aperta.Storage` behaviour: `put(key, binary_or_path, opts)`,
  `presigned_get_url(key, opts)`, `delete(key)`.
- Implement `Aperta.Storage.S3` using `ExAws.S3`.
- Tests: unit tests that hit a MinIO test bucket, guarded with a `@moduletag
  :storage` tag so CI can skip if MinIO is absent.

### Step 4 ŌĆö Auth scaffold, locked to one user

- `mix phx.gen.auth Accounts User users` (follow Phoenix 1.8 scoped auth
  prompts; use LiveView flow).
- Remove the `/users/register` route from the router and delete the
  registration LiveView + template (or leave the module and drop the route ŌĆö
  decide during the step). Login remains.
- In `priv/repo/seeds.exs`: create a single user from
  `APERTA_USER_EMAIL` / `APERTA_USER_PASSWORD` env vars (with safe dev
  defaults) if one doesn't exist.
- Tests: update/prune generator-produced tests so only login flows remain.

### Step 5 ŌĆö `documents` schema + `Aperta.Library` context

- Migration per the **Data model** section above.
- `Aperta.Documents.Document` schema + changeset.
- `Aperta.Library` with the functions listed above. `update_current_page/3`
  enforces LWW: takes a `client_updated_at`, only writes if newer than the
  stored one, returns `{:ok, doc}` either way (idempotent) so we never surface
  "stale" errors to the UI.
- Unit tests for the LWW rule (older timestamp ignored, newer wins, same-ts is
  a no-op).

### Step 6 ŌĆö Library LiveView with drag-and-drop upload

- Route: `live "/", LibraryLive, :index` under an authenticated
  `live_session`. (Move the generated `PageController` home out of the way.)
- `ApertaWeb.LibraryLive`:
  - `allow_upload(:pdf, accept: ~w(.pdf), max_file_size: 500_000_000,
    max_entries: 10)`.
  - Drag-and-drop dropzone + file picker. Use list-syntax Tailwind classes per
    the code rule.
  - `consume_uploaded_entries/3`: `Storage.put/3` ŌåÆ `Library.create_document/2`.
  - Title default = PDF XMP title if we can get one cheaply from the first few
    bytes, else `Path.rootname(filename)`. Author left `nil` in v1 if not
    trivially available. (Keep server-side PDF parsing optional; OK to ship v1
    with filename-only titles and an "Edit title" affordance later.)
  - Uses LiveView streams for the document list (`phx-update="stream"`).
  - Each row links `navigate` to the reader.
- Tests: upload a tiny fixture PDF, assert the row appears in the stream and
  the object exists in MinIO (storage-tagged test).

### Step 7 ŌĆö Reader LiveView + PDF.js viewer

- Vendor `pdfjs-dist` into `priv/static/vendor/pdfjs/` via an `npm install` in
  `assets/` plus a copy step in `mix assets.setup` (document it).
- Route: `live "/library/:id", ReaderLive, :show`.
- `ApertaWeb.ReaderLive`:
  - On `mount/3`: fetch the document (scoped to `current_scope.user`),
    generate a short-lived `Storage.presigned_get_url/2`, assign
    `@viewer_src` as `"/vendor/pdfjs/web/viewer.html?file=" <> URI.encode_www_form(url)`.
  - Template: full-height `<iframe>` with a colocated `.PdfViewer` hook that
    owns `phx-update="ignore"`.
- Colocated hook `.PdfViewer`:
  - On mount: listen for `message` events from the iframe; when the viewer
    reports `pagesloaded` with `numPages`, `pushEvent("pages_loaded", ...)`.
    When it reports `pagechanging`, debounce (~400ms) and
    `pushEvent("page_changed", %{page, client_updated_at})`.
  - `handle_event("sync_to", %{page})` from the server: `postMessage` into the
    iframe, which sets `PDFViewerApplication.page = page`.
- Server events:
  - `handle_event("pages_loaded", ŌĆ”)` ŌĆö persist `page_count` once, jump the
    viewer to the saved `current_page`.
  - `handle_event("page_changed", ŌĆ”)` ŌĆö call `Library.update_current_page/3`,
    broadcast `{:page_updated, page, origin_id}` on
    `"document:#{id}"` if it was accepted.
- On `mount/3` when `connected?(socket)`: subscribe to `"document:#{id}"`.
  `handle_info({:page_updated, page, origin})` ŌĆö if `origin != self_origin`,
  `push_event(socket, "sync_to", %{page: page})`.
- Tests: LiveView test that mounts the reader, asserts the iframe is present
  with the presigned URL, and that a simulated `page_changed` event updates
  the DB and broadcasts.

### Step 8 ŌĆö Cross-device sync integration test

- Two-LiveView test (same doc, two processes): `page_changed` from one
  process triggers a `sync_to` push on the other; the originating process does
  not receive its own echo.
- Verify the LWW rule end-to-end: a reply with an older `client_updated_at`
  does not move the page.

### Step 9 ŌĆö Polish pass

- Empty state on the library (no documents yet) using the streams-friendly
  Tailwind `hidden only:block` pattern.
- Delete button on each row (with confirmation), removes the row + MinIO
  object.
- Loading state on upload (spinner + progress), error toasts via
  `<Layouts.app flash={@flash}>`.
- Responsive: library grid collapses to a single column on small screens.
- Keyboard shortcut inside the reader: `Esc` navigates back to the library
  (delegated to PDF.js's own keybindings where possible).

### Step 10 ŌĆö README + run book

- Update `README.md`:
  - MinIO in the Quickstart (ports, default credentials).
  - `APERTA_USER_EMAIL` / `APERTA_USER_PASSWORD` for seeding the single user.
  - How to open the app, upload a PDF, and test cross-device sync locally
    (two browsers, same account).
- Run `mix precommit` one final time.
